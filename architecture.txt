DATA FLOW
---------
1. User activates license key → activate_license decrypts AES-256-GCM token,
   stores validated auth in settings table.
2. User configures devices (model, slave ID, COM port / IP, registers) →
   apply_bus_config validates and stores in EngineState.
3. User clicks Start → toggle_polling launches run_polling_loop as a Tokio task.
4. Polling loop reads Modbus registers → emits "meter-data" events (Tauri)
   and pushes JSON to WebSocket clients (web mode).
5. Frontend receives events → updates readings Map + 60-point history buffer →
   re-renders MetricCard grid and WaveformChart.
6. On fault → "meter-fault" + "status-changed" events → Toast + tab indicator;
   loop auto-retries after 2 s.
7. User requests export → export_to_excel queries meter_history, writes .xlsx
   via rust_xlsxwriter.
8. Intranet web server (Axum) lets phones/tablets on the same network use the
   same React SPA served from the embedded binary, tunnelling commands via REST.

DUAL-MODE DEPLOYMENT
--------------------
  Desktop (Tauri native):  invoke() / listen() IPC — zero network traffic
  Web/Intranet (browser):  fetch POST /api/<cmd>   — served by embedded Axum server
  Both modes share identical React frontend; api.ts abstracts the difference.

--------------

  - Token fields: created_at, duration_days, ttl_hours=1, username, project_name,
    allowed_meters, mode (offline/online), tier (1/2/3), protocols (RTU/TCP/All)
  - Activation window: 60 minutes from generation
  - Stored in DB settings table; expiry checked on every get_auth_state call
  - cloud_sync feature flag: enables SMTP alarm email + cloud API calls (optional build)
